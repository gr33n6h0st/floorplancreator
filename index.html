01 / The Interface & Standards

Fig 1.1 / Drafting Workspace
Mechanism: Constraint-Based Snapping

As illustrated in Fig 1.1, the software utilizes a "Smart Snap" engine. When a student drags the Square Room component (seen on screen) near another element, the software automatically aligns the vectors to 90° or 45° angles. This mimics the "Object Snap" (OSNAP) features found in industry-standard tools like AutoCAD and Revit, ensuring that students produce clean, water-tight geometry rather than messy sketches.

Standard: Metric Precision

The Evidence: The interface is designed to prevent "eyeballing." As shown by the dimension line in the screenshot (e.g., 834 cm x 525 cm), every wall has a defined numerical value.

The Learning: This fulfills Curriculum Expectation B1, which requires students to "create technical drawings using standard conventions." The student is forced to think in specific metric units, developing a sense of architectural scale.

02 / Visualization Engine

Fig 2.1 / 3D Walkthrough Mode
Pedagogical Value: Ergonomic Verification

Novice designers often struggle to translate 2D lines into 3D space. Fig 2.1 demonstrates the tool's solution: an instant 3D rendering engine. By toggling this view, students can perform Ergonomic Checks. They can verify if a window is too high, if a hallway feels claustrophobic, or if furniture blocks a path of travel. This immediate visual feedback creates a "Design-Test-Refine" loop, which is the core of the Technological Design process (Expectation A1).

03 / Implementation & Assessment

Fig 3.1 / Automated Data

Scenario A: Mathematical Constraints

Numeracy Integration

The Evidence: Fig 3.1 highlights the "Live Area" text inside the room (e.g., 55.15 m²).

The Learning: In the 'Residential Design' unit, students must adhere to strict total area limits. This feature provides mathematical scaffolding. As they move a wall, the number changes instantly. This forces a negotiation between aesthetic desires ("I want a huge living room") and numerical reality ("I only have 5m² left"), teaching resource management and adherence to project briefs.

Fig 4.1 / Code Compliance Check

Scenario B: Accessibility Standards (AODA)

Regulatory Law

The Evidence: Fig 4.1 verifies a wall opening of 201 cm (converted to 2010 mm), which significantly exceeds the standard door width.

The Learning: This demonstrates compliance with the Accessibility for Ontarians with Disabilities Act (AODA). While standard doors are often narrow, students verify that openings exceed the 860 mm barrier-free minimum. This ensures the design is wheelchair-accessible, moving students from abstract drawing to legally compliant "Universal Design".

Fig 5.1 / Cloud Collaboration

Scenario C: Industry Workflow (BIM)

Preparation for Work

The Evidence: Fig 5.1 captures the cloud-sharing menu, generating a unique file-type for the project.

The Learning: Modern architecture relies on Building Information Modeling (BIM), where files are live and shared among engineers, architects, and clients. This feature mimics that workflow. Students submit a variation of file-types rather than a printed paper, allowing the teacher (the client) to inspect the digital model remotely, reinforcing digital literacy and professional communication methods.

Fig 6.1 / Systems Integration

Scenario D: Interior & Mechanical Design

Systems Coordination

The Evidence: Fig 6.1 visualizes the complete interior environment, integrating furniture placement with mechanical spatial requirements.

The Learning: This demonstrates the intersection of Interior Design and Mechanical Design. Students move beyond empty shells to fully realized environments, ensuring that aesthetic choices (furniture) do not conflict with functional mechanical zones. This simulates real-world "Systems Integration."
